# Boxes animate their real size — Motion's layout-scale is banned

A card that changes size (an Entry row swapping to its inline editor, the add card's response zone) animates **its actual `height`, with contents swapped by a fade-through** (old fades out ~80 ms at the old size → the box springs to the new size, top-anchored → new fades in ~150 ms; the `FadeSwap` primitive in `src/screens/main/FadeSwap.tsx`) — never Motion's `layout` prop on a box whose size changes. `layout` interpolates size as a **scale transform**, which visibly stretches the card's corners, text, and controls mid-flight, and the app's motion vocabulary (spec § Motion) is "space and fade": boxes make space, content only fades, nothing ever distorts. `layout="position"` stays legal — translation cannot distort — and is what drives sibling reflow; during a height animation siblings need no animation at all, since document flow pushes them in perfect lockstep frame by frame. Cost accepted: animating height does real layout work every frame where layout-scale is a GPU-cheap transform — fine at this app's scale (one phone screen of cards), and the cheap alternative is the exact stretch the style exists to ban. This is deliberate, not naive Motion usage; don't "fix" it back to `layout`.
