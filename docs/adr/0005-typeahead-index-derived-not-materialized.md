# The typeahead index is derived, not materialized

The history typeahead's Suggestions (frecency ranking, normalized-label dedup keeping the freshest macros) are served from an **in-memory index built at runtime** from the entries listener — there is no `/history` or favorites collection in Firestore. A materialized collection would be a second write path that can drift (counts decremented on delete, drift on missed updates — V1's favorites table died exactly this way), and every frecency tweak would be a data migration; derived, the parameters tune freely and deletes simply fall out of the index. Cost accepted: a new device's first sync reads the full entry history once (~10k docs for years of use, well under Firestore's free-tier quotas), and the ~3-week frecency half-life means old history barely affects ranking anyway.
